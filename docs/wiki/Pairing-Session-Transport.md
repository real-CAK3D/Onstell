# Pairing Session Transport

This page defines the first trusted LAN session shape for Onstell. It is a design target only until the release gates in [Input Forwarding Architecture](Input-Forwarding-Architecture) are satisfied.

No Milestone 2 code should open sockets, forward input, sync clipboard data, or accept remote control traffic.

## Transport Choice

First prototype transport: WebSocket over TCP with Onstell-owned encrypted binary frames.

Why this first:

- Easy to inspect and run on Windows, macOS, Linux, and Raspberry Pi.
- Simple request/response control messages for pairing, heartbeat, and session setup.
- Works with local firewall prompts and common LAN debugging tools.
- Lets the app keep encryption, identity, replay checks, and protocol versioning in Onstell-owned message frames instead of relying on WebSocket as the security boundary.

Fallback path:

- If WebSocket libraries or firewall behavior become awkward, fall back to length-prefixed TCP using the same encrypted frame format.
- If latency, migration, or multiplexing becomes the limiting factor, evaluate QUIC after the WebSocket/TCP protocol proves the session model.

Production path:

- Production transport must be authenticated and encrypted before real routing starts.
- QUIC is the preferred later candidate for a lower-latency production transport, but it should reuse the same identity, trust, revocation, and message semantics.

## Device Identity

Each installed Onstell app should have a local identity keypair created on first run.

Trust records should store:

- Stable device id.
- User-visible device name.
- Peer public identity key.
- Public key fingerprint shown during pairing.
- First trusted timestamp.
- Last seen timestamp.
- Trust state: local, unpaired, pending, trusted, blocked.
- Protocol minimum and maximum versions accepted for that peer.
- Optional nickname.

Trust records are local decisions. If one machine revokes or blocks another, that local app must refuse sessions from the blocked peer even if the peer still considers itself trusted.

Milestone 3 defines this shape in `apps/desktop/src/pairingTrustStore.ts` as a fake local-only prototype. The identity record explicitly reports that key material is `not-created`; no private key, socket, pairing exchange, nonce, session key, or cryptographic frame is generated in this phase.

## Pairing Handshake

The first real pairing flow should be:

1. Devices discover each other on the LAN as untrusted candidates.
2. The user starts pairing on one device.
3. Both devices exchange ephemeral pairing messages and show the same short code.
4. The user confirms the code on both devices.
5. Each device stores the peer identity key and trust record.
6. The pairing channel closes.
7. Later routing sessions must authenticate against the stored trust record.

The pairing code must be derived from both devices' public identity material and fresh ephemeral handshake material. It must not be a random code generated by only one side.

## Session Handshake

A trusted session starts only after both devices prove identity and freshness.

Suggested message flow:

1. `hello`: device id, supported protocol versions, role, random nonce, and public identity key fingerprint.
2. Trust lookup: reject if the peer is missing, unpaired, pending, blocked, revoked, or below the minimum protocol version.
3. `challenge`: responder nonce and chosen protocol version.
4. Key agreement: derive an ephemeral session key using fresh key material from both sides.
5. Authentication: each side signs or authenticates the transcript with its stored identity key.
6. `session-ready`: both sides confirm the encrypted frame key, peer role, selected monitor metadata, and session id.
7. Routing may arm only after `session-ready` and the local routing trust gate both pass.

No pointer, keyboard, clipboard, file-transfer, or layout mutation message may be accepted before `session-ready`.

## Frame Rules

Encrypted session frames should include:

- Protocol version.
- Session id.
- Monotonic sequence number.
- Message type.
- Payload length.
- Payload.
- Authentication tag.

Every accepted frame must be bound to the active session id and the negotiated protocol version.

## Replay and Downgrade Protection

Replay protection:

- Reject repeated sequence numbers.
- Reject out-of-window sequence numbers.
- Reject frames from old session ids.
- Reject frames after release, disconnect, trust loss, or revocation.

Downgrade protection:

- Store a minimum accepted protocol version in each trust record.
- Include supported versions in the authenticated handshake transcript.
- Reject peers that try to negotiate below the stored minimum.
- Require explicit user action before lowering a trusted peer's minimum protocol version.

## Blocked and Revoked Devices

Blocked or revoked devices cannot reconnect silently.

Rules:

- A blocked trust record always wins over discovery.
- A blocked peer must not become pending automatically.
- A revoked peer must return to unpaired or blocked state and lose active sessions immediately.
- Any active session from a now-blocked peer must trigger release behavior.
- Re-pairing a blocked device requires explicit user action from the local app.

## Reconnect Expectations

Reconnect may be automatic only for trusted, non-blocked peers.

Reconnect must:

- Create a fresh session id.
- Use fresh nonces and ephemeral key material.
- Re-run the trust lookup.
- Re-run protocol version checks.
- Start in an armed or local state, never directly in forwarding.
- Release all held input state from the previous session before accepting new routing messages.

Reconnect must not:

- Reuse old session keys.
- Resume unacknowledged input events.
- Bypass the widget's visible target state.
- Reconnect blocked, revoked, pending, unknown, or unpaired devices.

## Message Families

Early LAN prototype messages:

- `session.hello`
- `session.challenge`
- `session.ready`
- `session.heartbeat`
- `session.release`
- `routing.arm`
- `routing.pointer_delta`
- `routing.pointer_button`
- `routing.key_code`
- `routing.pause`
- `routing.disconnect`

Clipboard, file transfer, device settings, and update messages are out of scope for the first session prototype.

## Failure Handling

On any authentication, replay, downgrade, trust, or permission failure:

- Stop accepting routing messages from that session.
- Clear the active target.
- Release all tracked held key and pointer-button state.
- Show a disconnected or blocked state in the widget.
- Record only sanitized diagnostic metadata.

Do not show raw keys, clipboard data, typed text, tokens, session keys, nonces, or message payloads in logs.

## Open Questions

- Which Rust crypto/session crate should carry the first authenticated frame implementation?
- Should discovery advertise only device ids and names, or also identity-key fingerprints?
- Should a trusted peer be allowed to rotate identity keys through an explicit recovery flow?
- Should QUIC become the default before public beta, or remain a later performance improvement?
